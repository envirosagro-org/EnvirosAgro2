
# EnvirosAgro Quantum-Ready Database Schema (Firestore)

This document outlines the proposed NoSQL database schema for the EnvirosAgro ecosystem, designed to be scalable, resilient, and "quantum-ready." This means the data is structured to be efficiently processed by future quantum algorithms for optimization, analysis, and cryptography.

The database will source all signals, data, and information from the EnvirosAgro Network (Stewards, Nodes, Shards) and synchronize with the Farm OS.

---

## Core Principles

1.  **Decentralized & Immutable:** Data is generated by a distributed network. We treat data shards as immutable events, chaining them together to form a verifiable history.
2.  **Signal-Driven:** The entire system is reactive. "Signals" are first-class citizens that trigger workflows, update states, and communicate between nodes.
3.  **Quantum-Ready:** Data is structured in high-dimensional vectors and time-series where applicable, making it suitable for analysis by Quantum Machine Learning (QML) models to identify patterns in sustainability, crop health, and economic activity. Quantum optimization algorithms could be used to solve complex logistical problems, like optimizing resource allocation across the network.

## Main Collections

### 1. `/stewards/{stewardId}`
Stores the profile and aggregated data for each human operator in the network.

-   **`esin`**: `string` (e.g., "EA-ABCD-WXYZ") - *Immutable Social ID*
-   **`name`**: `string`
-   **`email`**: `string`
-   **`role`**: `string` (e.g., "REGENERATIVE FARMER")
-   **`m_constant`**: `float` - *The node's resilience score, calculated periodically.*
-   **`wallet`**: `map` (Contains balances, tier, etc.)
-   **`linked_nodes`**: `array` (List of `nodeId`s managed by this steward)
-   **`location`**: `geopoint`

### 2. `/nodes/{nodeId}`
Represents a physical hardware unit (e.g., `EOS-HW-001`) in the field.

-   **`stewardId`**: `string` (Owner of the node)
-   **`hardware_type`**: `string` (e.g., "EOS-HW-001")
-   **`status`**: `string` (e.g., "ONLINE", "SYNCING", "STRESSED")
-   **`location`**: `geopoint`
-   **`last_heartbeat`**: `timestamp`
-   **`telemetry_summary`**: `map` (Aggregated real-time data, e.g., avg_temp, avg_moisture)

#### Sub-collection: `/nodes/{nodeId}/telemetry_shards`
Immutable, time-series data sent from the node. This is the raw material for calculating the `m-Constant`.

-   **`/telemetry_shards/{shardId}`**
    -   **`timestamp`**: `timestamp`
    -   **`data`**: `map` (e.g., `{ "soil_ph": 6.8, "temp_c": 25, "spectral_hash": "b4a..." }`)
    -   **`zk_proof`**: `string` - *Zero-Knowledge proof of data integrity.*

### 3. `/signals/{signalId}`
A centralized collection for all system-wide communications, alerts, and actions.

-   **`origin`**: `string` (e.g., "FarmOS", `nodeId`, `stewardId`)
-   **`target`**: `string` (`nodeId`, `stewardId`, "ALL")
-   **`type`**: `string` (e.g., "STRESS_ALERT", "SYNC_REQUEST", "AUDIT_REQUEST")
-   **`payload`**: `map` (Data associated with the signal)
-   **`status`**: `string` ("PENDING", "ACKNOWLEDGED", "COMPLETED")
-   **`timestamp`**: `timestamp`

### 4. `/carbon_credits/{creditId}`
Represents a digitally minted carbon credit.

-   **`stewardId`**: `string`
-   **`nodeId`**: `string` (Originating node)
-   **`mint_date`**: `timestamp`
-   **`status`**: `string` ("MINTED", "AUDITED", "SOLD")
-   **`audit_trail`**: `array` (List of audit events and proofs)
-   **`value_tons_co2e`**: `float`

## Farm OS Synchronization

Synchronization with the Farm OS will be handled via a dedicated API endpoint (e.g., a Firebase Cloud Function).

1.  **Farm OS -> Database:**
    -   The Farm OS generates operational data (e.g., planting schedules, harvest yields).
    -   It posts this data as a `FARM_OS_UPDATE` signal to the API endpoint.
    -   The Cloud Function processes this signal, updates the relevant `/nodes` or `/stewards` documents, and may generate new signals for the network (e.g., a `HARVEST_READY` signal).

2.  **Database -> Farm OS:**
    -   The Farm OS can query the database via the same API for real-time telemetry data, steward information, or network-wide alerts. This allows it to adjust its internal logic based on live conditions from the EnvirosAgro network.

This schema provides a robust foundation for building the EnvirosAgro platform. The next step would be to start implementing these collections and the services that interact with them.
